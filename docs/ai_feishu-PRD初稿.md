# 【PRD初稿】飞书原生本地 AI 知识库 (AI_Feishu) - Phase 1 MVP

## 1. 产品定位与核心价值
*   **产品名称**：AI_Feishu (暂定名)
*   **核心定位**：100% 依托飞书平台的轻量级、本地化多模型 AI 助理与知识库引擎。
*   **核心价值**：消除“多 AI 工具切换”与“飞书文档搬运”的摩擦力；提供飞书内原生级别的极致对话体验；数据与向量库 100% 本地化运行，保障绝对隐私。
*   **真理源设定**：飞书 IM 为唯一交互入口与聊天记录载体；飞书云文档为唯一高质量知识源；本地只保留极简配置文件与无头向量库。

## 2. 核心架构与系统边界 (系统设计约束)
本系统采用 B/S 架构本地私有化部署（后续演进为 Tauri 桌面端）。
*   **前端控制台 (Web Admin UI)**：纯 React/Vite 构建。**绝对禁止包含任何聊天交互界面**。仅作为系统配置、大模型 Key 管理、知识库状态监控、服务监控的后台面板。
*   **本地大核引擎 (Local Core Engine)**：Node.js/TS 守护进程。通过 `lark-oapi` 维持 WebSocket 长连接；通过 `Vercel AI SDK` 统一路由各大模型厂商。
*   **数据持久层 (Data Layer)**：禁止使用任何独立进程数据库（如 Postgres/Milvus）。采用 `LanceDB` 作为本地无头向量库，采用 `SQLite` 存储轻量系统配置。

## 3. MVP 核心功能模块划分

### 模块一：本地中控台面板 (Local Web Admin)
*目标：提供极简的可视化系统配置，脱离代码修改。*
1.  **全局环境配置**：配置飞书机器人的 App ID 与 App Secret。
2.  **模型路由管理**：配置多 LLM（如 OpenAI、Anthropic、Gemini、Ollama 等）的 API Key 和 Base URL。支持设置“系统默认全局模型”。
3.  **知识库基准绑定**：提供一个输入框，要求用户输入**一个或多个飞书云文档文件夹的 URL**，作为本地向量库监控和同步的“指定知识源”。
4.  **状态监控台**：可视化展示当前 WebSocket 连接状态、已向量化的知识库 Chunk 数量。支持手动点击“全量同步/刷新知识库”按钮。

### 模块二：飞书端内原生交互 (IM Native Interaction)
*目标：100% 在飞书完成所有 AI 交互，体验比肩官方应用。*
1.  **卡片式会话启动器 (Session Starter)**：
    *   触发：用户点击飞书底部自定义菜单。
    *   响应：系统下发一张“新建对话”飞书交互卡片 (Interactive Card)。
    *   卡片元素：包含一个【选择 AI 引擎】的下拉菜单（动态拉取中控台已配置的模型列表），以及一个【开始对话】按钮，以及对话主题和描述编辑。
2.  **话题级隔离 (Thread-based Session)**：
    *   用户在启动卡片中选择模型并回复后，**该 Thread (话题) 永久锁定该大模型**。
    *   所有在此 Thread 下的回复（Reply in Thread）都共享同一个历史上下文。
    *   跨 Thread 上下文严格隔离。
3.  **伪流式打字机输出 (Stream Rendering)**：
    *   后端接收大模型流式（SSE）返回，通过飞书 `Patch API` 高频（如每 200ms-500ms）更新同一张消息卡片或富文本消息，实现类似 ChatGPT 的视觉打字效果。并完美渲染 Markdown 格式（代码块、加粗等）。

### 模块三：AI 工具调用与飞书联动 (Tool Calling & Feishu API)
*目标：利用大模型原生 Tool Calling 能力，赋予 AI 读写飞书的能力，无需硬编码指令按钮。*
后端向 LLM 注册以下三个核心 Tools：
1.  **即时链接阅读 (`read_feishu_url`)**：
    *   场景：用户在聊天框直接发送包含飞书云文档/多维表格链接的文本（例：“总结这个链接的内容 https://…”）。
    *   逻辑：AI 判断需调用此 Tool，后台截获并调用飞书 API 拉取该文档内容，清洗为 Markdown 后作为上下文注入系统，LLM 据此生成回答。
2.  **历史知识检索 (`search_local_kb`)**：
    *   场景：用户提问涉及历史沉淀的知识（例：“我们上个月确定的 Q3 目标是什么？”）。
    *   逻辑：AI 判别需查阅资料，调用此 Tool，后台在本地 LanceDB 中进行语义检索，返回 Top-K Chunk 文本给 LLM 进行 RAG 回答。
3.  **对话成果归档 (`save_to_new_doc`)**：
    *   场景：用户下达归档指令（例：“把我们刚才探讨的架构方案整理成文档保存” 或发送 `/save`）。或者对话结束后发送询问是否归档卡片，用户可以选择性归档，或者点击总结，进行多红方式归档
    *   逻辑：后台提取当前 Thread 完整聊天记录，LLM 根据记录生成结构化 Markdown。后台调用此 Tool (映射飞书创建文档 API)，在指定的“AI 沉淀”文件夹中创建**全新的飞书云文档**，并回复该新文档的卡片链接。
    *   约束：**绝对不修改任何已有文档的内容**。

### 模块四：本地极简 RAG 流水线 (Local RAG Pipeline)
*目标：轻量级、无感知的文档向量化。*
1.  **定向拉取与清洗**：基于【模块一】配置的飞书文件夹 URL，后台调用飞书目录 API，遍历其下属所有文档，拉取纯文本内容。
2.  **本地分块与向量化 (Chunking & Embedding)**：使用极简的 TextSplitter 算法（如按段落或 Token 分割），调用用户配置的轻量 Embedding 模型（如 API 或本地模型）。
3.  **本地落库**：写入本地 Node.js 驱动的 LanceDB 文件夹。包含字段：`id`, `vector`, `text_chunk`, `doc_url`, `doc_title`。

## 4. 关键用户交互流程图 (User Flows)

**Flow A: 创建特定模型对话**
用户在飞书单聊中发送 `/start` 或者点击菜单按钮-> 收到启动卡片 -> 下拉选择 "Claude-3.5" -> 在该卡片下点击“回复”发送消息 -> 后台识别该 Thread 绑定 Claude-3.5 -> 调用大模型 -> 流式更新飞书消息卡片回复用户。

**Flow B: 无缝文档问答 (On-the-fly RAG)**
用户在已有 Thread 中输入：“帮我提取这篇文章的核心观点 [飞书文档链接]” -> 后台 LLM 触发 Tool Calling `read_feishu_url` -> 后端调用飞书 API 获取该文档文本并返回给 LLM -> LLM 总结并流式输出到飞书。

**Flow C: 文档检索**
用户在当前 Thread 输入：最近的会议记录中关于--的讨论和进度总结并发送， -> LLM 提取 Thread 上下文，检索知识库 -> LLM -> AI 在聊天框回复。

**Flow D: 知识沉淀闭环**
用户认为当前 Thread 讨论极具价值，回复：“把以上讨论提炼成落地计划并存档” -> LLM 提取 Thread 上下文，整理出标准 Markdown -> 触发 Tool Calling `save_to_new_doc` -> 后端在飞书指定目录创建新文档 -> AI 在聊天框回复：“✅ 已归档，点击查看：[新飞书文档链接卡片]”。


## 5. 明确不在一期范围内的功能 (Out of Scope for Phase 1)
为了保证 MVP 的快速落地与极高可用性，以下功能**严禁**出现在一期设计和开发中：
1.  **Web 端的聊天界面**（彻底阻断双端同步复杂性）。
2.  **飞书聊天记录自动向量化**（避免“脏数据”污染知识库，必须由用户手动确认，并通过 Tool Calling 显式归档）。
3.  **覆盖修改/插入现有的飞书文档**（避免破坏用户原有的复杂富文本排版和协同冲突）。
4.  **复杂的图谱可视化 (Karpathy Graphify)**（图谱渲染延后至 Phase 2 桌面端规划）。
5.  **多 Agent 工作流编排**（如：先查数据、再写代码、再发邮件的复杂节点链路，本期仅依靠 LLM 原生 Tool Calling 的单步决策）。

# 【PRD 补充文档】全面拥抱飞书官方 MCP 协议体系

**补充背景说明**：
鉴于飞书官方已正式发布官方 MCP (Model Context Protocol) Server，系统核心架构中“AI 与飞书交互”的底层实现方式将从“自研 Tool Calling 封装”全面升级为“对接官方 MCP 标准协议”。此举将节省 80% 的 API 对接开发量，并获得官方最高级别的数据解析支持。

## 补充 1：修改【2. 核心架构与系统边界】中的相关定义
*   **增加通信层定义**：本地大核引擎 (Local Core Engine) 将引入 `MCP Client` 模块。系统将不再通过传统 `lark-oapi` 手写繁琐的富文本清洗逻辑，而是直接将飞书官方提供的 MCP Server 挂载到大模型（如 Vercel AI SDK）的可用工具集中。
*   **关于 Feishu CLI 的取舍 (系统边界约束)**：虽然官方提供了 Feishu CLI，但考虑到本系统以 Node.js 常驻进程为核心，为保证内存管理与并发性能，**一期 MVP 将全面采用 MCP 协议进行通信，禁止使用 Node.js 子进程 (child_process) 去调用 Feishu CLI**，避免产生不必要的系统性能损耗。

## 补充 2：重构【模块三：AI 工具调用与飞书联动】的底层实现方式
*（原模块三的业务场景不变，但底层执行逻辑替换为 MCP 标准化流程）*

1.  **不再维护自定义 Tools，全盘映射官方 MCP 能力**：
    系统启动时，Node.js 后端连接飞书官方 MCP Server，并直接透传（Proxy）其提供的标准化 `Resources` 和 `Tools` 给底层 LLM。
2.  **核心场景对应的 MCP 官方能力映射**：
    *   **即时链接阅读 (映射官方 MCP 读能力)**：当用户发送飞书文档 URL 时，系统不再自写解析器，直接调用飞书官方 MCP 暴露的 `read_document` 工具。官方 MCP 会返回经过最优化排版的 Clean Markdown，直接喂给 LLM。
    *   **历史知识检索 (混合 RAG 架构微调)**：如果飞书 MCP Server 直接提供了 `search_wiki_or_drive` (飞书全局搜索) 能力，系统将优先采用官方搜索结果作为 RAG 的一级召回源；本地 LanceDB 则作为针对特定目录深度切分的二级召回源。
    *   **对话成果归档 (映射官方 MCP 写能力)**：大模型生成 Markdown 后，直接调用官方 MCP 的 `create_and_write_document` 工具，彻底规避传统 API 可能引发的 Block 节点转换错误。

## 补充 3：新增 MVP 一期配置项：【MCP 鉴权与 Server 管理】
*(需补充至“模块一：本地中控台面板”)*
1.  **官方 MCP Server 接入配置**：在中控台中新增“Feishu MCP 配置”模块，支持用户一键启动/配置本地侧或远端托管的飞书 MCP Server 参数（如 Token、运行端口等）。
2.  **可用工具授权开关**：提供可视化开关，允许用户勾选大模型可以调用的飞书官方 MCP 能力（例如：允许 AI 读取文档 ✅，允许 AI 自动建群 ❌，允许 AI 发送邮件 ❌），贯彻“只读+新建”的 MVP 安全原则，防止 AI 越权操作。
